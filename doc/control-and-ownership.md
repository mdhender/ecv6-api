# Control and Ownership

The domain model for *who controls and owns what* in EC: the
account → player → controller → faction → asset chain. This is design in
progress — settled parts are stated plainly; **open threads are flagged at the
end**. Companion to [architecture.md](architecture.md) (the two domains) and
[model.md](model.md) (concept ↔ type ↔ schema mapping).

## The boundary, by identifier

The application and game domains meet at a small set of identifiers. What each
domain may see is the whole ballgame:

| Identifier | Domain | Engine's relationship |
| --- | --- | --- |
| `account_id` | application | **unaware** — the engine never sees it |
| `game_id` | game | **consumes** (given to it) |
| `player_id` (PK of `game_account_role`) | game | **consumes** (given to it) |
| `faction_id` | game | **owns** — creates and mutates it |

`game_account_role` is therefore a **boundary table**: its `account_id` column is
application-only, while its PK (`player_id`) and `game_id` are game-consumable.
The engine knows a faction is controlled by `player_id = 42`; it *cannot* resolve
42 to a human, email, or login. So "the engine is unaware of the application
side" means precisely **unaware of accounts, not of players**.

## Control is not ownership

This is the distinction the whole model rests on:

- **Controllers control. Factions own.** A player (or NPC) *controls* a faction;
  the *faction* owns the assets. A player never owns anything.
- Because assets hang off the faction, **a player leaving cuts one link
  (control) and touches nothing beneath it.** The faction and every asset persist.

## Controllers

A **controller** is an agent that issues orders each turn for the faction(s) it
controls. There are two kinds:

- **player** — a human. Orders arrive via the API and are collected
  application-side; the engine sees them tagged by `player_id`.
- **npc** — an engine agent. Orders are generated by the engine itself. The
  default agent issues the bare-minimum orders to feed the population and keep the
  lights on.

The payoff: **the engine's turn loop does not branch on human-vs-NPC.** It needs
one order-set per controlled faction and does not care where it came from — player
orders handed in, NPC orders generated. One code path, and account identity never
reaches the engine even during order submission.

- Each faction has **exactly one** controller (never orphaned).
- A controller controls **zero-or-more** factions.

## Factions

A **faction** is the persistent in-game owner (the engine owns `faction_id`).

- **Created after a player joins**, when the account calls a setup-ritual API
  endpoint. The engine mints the `faction_id` and binds it to the caller's
  `player_id`. (A just-joined player controls **zero** factions until setup runs.)
- **Never dies.** A faction may be reduced to zero assets and still continues in
  game state.
- **Never orphaned.** It always has exactly one controller. When a player leaves
  (application side: the seat's `is_active` goes false), the engine assigns an
  **NPC** to control the faction.
- **A player controls `0..*` factions.** The `0` is real and *permanent*, not only
  transient: the **GM seat commands no faction at all**, and a just-joined player
  has none until setup. **The engine must never assume a 1:1 player↔faction** — the
  GM is the standing exception. The upper `*` (multi-faction) will not be
  implemented for a while — treat it as ~one per non-GM player in the near term.

## NPCs

An **NPC** is a controller whose orders come from an engine agent rather than a
human. Engine-owned. We are not committing to an implementation; the intent is
agents that issue orders each turn, with the **default agent** doing minimal
upkeep (feed population, keep the lights on).

- **"Independent ships and colonies"** (the rulebook's term) are simply the assets
  in an NPC-controlled faction.
- **Initially one NPC** — the "mother duck," keeping everyone alive and warm —
  controls all independents with the minimal agent.
- Later implementations may add flavored NPCs (pirates, zealots, expansionistas).
  No constraints beyond the minimal agent.
- NPCs are **engine-internal** and not player-facing, so they need not surface in
  the player docs.

## Assets: the entity chain

- **ship_or_colony** — a game entity owned by **exactly one** faction (never
  un-owned). Transferable: ownership can change hands between factions. A faction
  owns `0..*` of them. **A colony is essentially a ship that never moves** — the
  two share the same shape (population, mandatory inventory) and differ in only a
  few important ways. (Whether that's one table with a type discriminator or two
  sharing structure is a schema question for later.)
- **population** — `0..*` per ship/colony; each belongs to one.
- **inventory** — `1..*` per ship/colony; each belongs to one. Inventory is
  **mandatory** for both ships and colonies: a ship without a hull is not a ship,
  and a colony carries the same floor.

Because assets belong to a faction and can be transferred, **ownership is a
turn-timebound fact** (see
[storing-state-as-timebound-facts.md](storing-state-as-timebound-facts.md)): a
transfer closes the old owner's interval and opens the new owner's.

## Cardinality summary

```
controller ─||──o< faction ─||──o< ship_or_colony ─┬──o< population
   │                                               └──|< inventory
   ├─ player : orders via the API (game-side player_id; account is app-only)
   └─ npc    : orders from an engine agent (default = minimal upkeep)
```

| Relationship | Reading |
| --- | --- |
| controller : faction | one controller controls `0..*`; each faction has exactly one controller |
| faction : ship_or_colony | `0..*`; each ship/colony owned by exactly one faction |
| ship_or_colony : population | `0..*`; each population in exactly one |
| ship_or_colony : inventory | `1..*`; each inventory in exactly one; every ship/colony has ≥1 |
| player : faction | a player (a kind of controller) controls `0..*` factions |

## Invariants

- **Nothing controllable is un-owned.** Every faction has one controller; every
  ship/colony has one faction.
- **Factions never die and are never orphaned.**
- **The engine sees `player_id`, never `account_id`.**
- **One owner per asset at a time**, but ownership is transferable across turns.
- **A player may command zero factions** — the GM seat always, a not-yet-founded
  player until setup. Never assume 1:1 player↔faction.

## Open threads

- **Order flow / the turn loop** — how player orders (collected application-side)
  and NPC orders (generated engine-side) are assembled and resolved each turn.
  This is the app ↔ engine hand-off, and it is **still open**.
- **Multi-faction control per player** — modeled as `0..*`, not implemented in the
  near term.
- **Ship vs. colony schema** — a colony is a ship that never moves; the few
  distinctions and whether they share one table are a schema question for later.
- **Flavored NPCs** (pirates / zealots / expansionistas) — future work.
- **Governments** — the docs have a faction govern through one or more
  *governments* (empires, republics, houses). A government provides **bonuses and
  penalties**; exact values are to be documented later, alongside rebels and
  insurrections. Where "government" sits in our model — a layer between faction and
  its holdings, or a faction attribute — is open.

## Relationship to the player-facing docs

The docs used a single word, "player," for what the server splits three ways. Both
the **account / player / faction** vocabulary *and* the faction **lifecycle**
(founding, independence, persistence) are now **adopted upstream** (ecv6-docs
`account.md`, `players.md`, `faction.md`). The docs describe independents as
something players encounter, but not *how* they run — `controller`, `npc`, and
`ship_or_colony` stay engine-internal and are deliberately kept out of the player
docs.
